1. Getting Started

Prerequisites

  • Node.js 18+ (uses fs.cpSync which requires Node 16.7+)
  • TypeScript 5.6+

Setup

git clone <repo-url>
cd pi-extension-creator
npm install            # Installs deps + sets up pre-commit hook
npm run build          # Compile TypeScript → dist/
npm test              # Run all 41 tests

Available Scripts

CommandDescription
npm run buildCompile TypeScript (tsc -p tsconfig.json)
npm testRun all tests (vitest run)
npm run test:watchWatch mode
npm run test:coverageWith coverage report
npm run validate:srcType-check without emitting
npm run precommitType-check + test (runs automatically on commit)
npm run cleanRemove dist/
npm run bootstrapBuild + self-install into pi

Pre-commit Hook

A pre-commit hook is installed by simple-git-hooks during npm install. It runs tsc --noEmit && vitest run before every commit.

Tip: If the hook fails, your commit is blocked until all type errors and test failures are fixed.

3. Module Reference

src/extension.ts
~100 lines — 8 imports, 1 tool definition, 1 export

Responsibility

Register the extension_creator tool with pi's defineTool() and coordinate the full lifecycle.

Key behavior

  • Normalizes mode/stage via router.ts
  • Runs validation for validate, review, install, update modes
  • Generates plans for plan/scaffold modes via planner.ts
  • Triggers install for install mode via installer.ts
  • Generates docs for document mode via documenter.ts
  • Formats response via renderer.ts

Exports

extensionCreator(pi: ExtensionAPI): void — default export, registers the tool

src/router.ts
~90 lines — 4 exported functions, 2 types

Responsibility

Infer the lifecycle stage and starting mode from natural language requests. This is the "brain" that translates user intent into structured actions.

Exports

  • normalizeMode(value, stage, goal, path?) → Mode — infers mode from stage, goal keywords, or path
  • normalizeStage(value, goal, path?) → Stage — infers stage from goal keywords or path
  • modeFromStage(stage) → Mode | undefined — maps stage to default mode
  • inferKindFromGoal(goal) → string | undefined — detects extension type
  • Mode type: 'plan' | 'scaffold' | 'review' | 'document' | 'validate' | 'install' | 'update' | 'remove'
  • Stage type: 'idea' | 'draft' | 'workspace' | 'validated' | 'installed' | 'maintenance' | 'unknown'

Inference Priority

  1. Explicit mode/stage parameter
  2. Stage-based default (idea → plan, draft → scaffold, etc.)
  3. Goal keyword matching
  4. Path presence → review
  5. Fallback → plan
src/planner.ts
~40 lines — 1 exported function, 1 interface

Responsibility

Generate structured build plans for new extensions.

Exports

  • buildPlan({ goal, extensionKind, strict, note }) → ExtensionPlan
  • ExtensionPlan — { title, summary, files, steps, cautions }
src/validator.ts
~280 lines — 1 exported async function, ~12 helper functions

Responsibility

Validate extension packages against pi's architecture rules. This is the quality gate.

Key behaviors

  • Reads and parses package.json
  • Resolves the declared entrypoint (pi.extensions → pi.entrypoint → main)
  • Enforces kebab-case package naming
  • Validates tsconfig.json exists
  • Runs tsc --noEmit with the project's tsconfig
  • Parses compiler output (handles both structured and free-form errors)
  • Scans entrypoint imports for undeclared dependencies
  • Batches all diagnostics — no early returns

Export

validateExtensionProject(packagePath: string) → Promise<ValidationResult>

src/installer.ts
~90 lines — 1 exported function, 1 internal function

Responsibility

Copy a validated extension package to ~/.pi-extensions/<name> deterministically.

Key behaviors

  • Refuses to install if validation status is fail
  • Uses package name from validation result for install directory
  • Uses fs.cpSync() for fast recursive copy
  • Replaces existing installation automatically
  • Updates ~/.pi/agent/settings.json
  • Logs settings.json errors via console.warn

Export

runDeterministicInstall({ sourcePath, validation }) → InstallResult

src/documenter.ts
~260 lines — 1 exported async function, ~12 internal helpers

Responsibility

Analyze extension source code and generate README.md and ARCHITECTURE.md automatically.

Key behaviors

  • Reads package.json for metadata
  • Resolves entrypoint from manifest (supports pi.extensions, pi.entrypoint, main)
  • Detects extension kind by scanning source and description
  • Detects commands (registerCommand), tools (registerTool), prompts, skills
  • Generates README with badges, features, install guide, dev commands
  • Generates ARCHITECTURE.md for complex extensions
  • Writes files directly to the source directory

Export

generateDocumentation(sourcePath: string) → Promise<GeneratedDocs>

src/renderer.ts
~190 lines — 3 exported functions, 2 interfaces

Responsibility

Convert internal ToolPayload into user-facing output and determine the next action.

Exports

  • renderPayload(payload) → string — full text rendering
  • chooseNextAction(...) → string — determine what user should do next
  • buildDocumentationGuidance(...) → DocumentationGuidance — legacy fallback
  • ToolPayload — all fields returned by the tool
  • DocumentationGuidance — documentation metadata
src/cli.ts
~140 lines — 1 main function, 2 helpers

Responsibility

Provide a standalone command-line interface.

Commands

  • validate [path] — Validate an extension package
  • review [path] — Same as validate
  • bootstrap — Build, validate, and install self
  • install-self — Same as bootstrap

Flags

--help, --version, --json, --verbose

4. Data Flow

Validation Flow

path → validator.ts │ ├── 1. Read package.json │ ├── missing → manifest.missing (error) │ └── invalid → manifest.invalid-json (error) │ ├── 2. Check package name │ ├── missing → manifest.name.missing (error) │ └── not kebab → manifest.name.not-kebab-case (error) │ ├── 3. Resolve entrypoint │ ├── pi.extensions[0] (preferred) │ ├── pi.entrypoint (fallback) │ ├── main (legacy, warning) │ └── none of the above → manifest.entrypoint.missing (error) │ ├── 4. Verify entrypoint file exists │ └── missing → entrypoint.missing (error) │ ├── 5. Check entrypoint naming │ ├── index.ts → entrypoint.index-file (warning) │ └── mismatched name → entrypoint.named-file (warning) │ ├── 6. Scan imports for undeclared deps │ └── found undeclared → dependencies.undeclared (warning) │ ├── 7. Check tsconfig.json │ └── missing → tsconfig.missing (error) │ ├── 8. Run tsc --noEmit │ ├── compiler not found → compiler.not-found (error) │ └── TS errors/warnings → parsed diagnostics │ └──→ ValidationResult { status, errors, warnings, details }

Install Flow

path → installer.ts │ ├── 1. Check validation.status !== 'fail' │ └── fail → return InstallResult { success: false } │ ├── 2. Resolve name: validation.packageName || path.basename(sourcePath) │ ├── 3. Check source exists │ └── missing → return InstallResult { success: false } │ ├── 4. Ensure ~/.pi-extensions/ exists │ ├── 5. Remove existing ~/.pi-extensions/<name> if present │ ├── 6. fs.cpSync(source, target, { recursive: true, force: true }) │ ├── 7. Update ~/.pi/agent/settings.json │ └── failure → console.warn (non-blocking) │ └──→ InstallResult { success, extensionName, installPath, message }

Document Flow

path → documenter.ts │ ├── 1. Read package.json (name, description) ├── 2. Resolve entrypoint source file ├── 3. Read entrypoint source ├── 4. Detect extension kind (tool, command, prompt, etc.) ├── 5. Detect commands (registerCommand), tools (registerTool) ├── 6. Check for prompts/ skills directories ├── 7. generateReadme() → string ├── 8. If complex: generateArchitecture() → string ├── 9. writeFileSync(README.md) └── 10. writeFileSync(ARCHITECTURE.md) (optional) └──→ GeneratedDocs { readme, architecture, filesWritten }

6. Extending the System

Adding a New Validation Check

  1. Add the check logic in src/validator.ts — push to errors[] or warnings[]
  2. Use a descriptive error code (e.g., manifest.*, entrypoint.*, tsconfig.*, compiler.*)
  3. Add a test in src/__tests__/validator.test.ts

Adding a New Mode

  1. Add the mode string to Mode type in src/router.ts
  2. Add inference logic in normalizeMode() in src/router.ts
  3. Add handling in src/extension.ts execute() body
  4. Add next-action logic in src/renderer.ts chooseNextAction()
  5. Add tests in src/__tests__/router.test.ts

Modifying the Documentation Generator

  1. Edit src/documenter.ts — generateReadme() or generateArchitecture()
  2. The readme/architecture are generated as pure function strings
  3. If adding a new detection, update detectExtensionKind() and the detection helpers

Modifying the Installer

  1. Edit src/installer.ts
  2. The install path is always ~/.pi-extensions/<packageName>
  3. Settings.json updates are non-blocking (errors are logged, not thrown)

Coding Standards

  • TypeScript strict mode enabled in tsconfig.json
  • ES2022 target, Node16 module resolution
  • Prefer named exports (except for extension entrypoint which uses default export)
  • No any unless absolutely necessary (pi SDK types use any in shims)
  • Tests should be deterministic and use temp directories
  • Add JSDoc comments for exported functions

    • 7. Design Decisions

    Why fs.cpSync() instead of custom recursive copy?

    Node 16.7+ provides native recursive cpSync. It's faster, handles edge cases (symlinks, permissions), and removes 19 lines of manual recursion from the codebase.

    Why batch diagnostics instead of early returns?

    One-pass validation reveals all issues at once, reducing the fix-revalidate loop from N iterations to 1. The user sees every problem in a single run.

    Why console.warn instead of silent catch?

    Silent error masking hides real problems (corrupt settings.json, permission issues). Logging gives debuggability without blocking installation.

    Why 8 modules instead of 1?

    The original extension.ts was ~400 lines handling mode inference, plan building, rendering, next-action logic, and documentation guidance — violating single responsibility. Splitting into 8 modules reduces the entrypoint to ~100 lines and makes each module independently testable.

    Why TypeScript over JavaScript?

    TypeScript provides type safety, better IDE support, and catches errors at compile time. The validator also runs tsc --noEmit as part of validation, ensuring extensions are type-safe before install.

    Why deterministic file copy instead of symlinks?

    File copies are explicit, portable, and don't break when the source is moved or deleted. The installer ensures pi always has a self-contained copy.
