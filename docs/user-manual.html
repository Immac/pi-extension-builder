1. Overview

The pi-extension-creator is a factory for pi extensions. It provides a complete lifecycle management system for building, validating, and installing pi extensions through LLM-driven tool calls.

Instead of manually managing extension files and running shell commands, LLMs use the single extension_creator tool to plan, scaffold, validate, document, and install pi extensions.

🎯 Intelligent Routing

Infers lifecycle stage and starting mode from natural language — no need to specify every parameter.

✅ One-Pass Validation

Batches all diagnostics before returning. See every error at once instead of fixing one at a time.

📦 Deterministic Install

Copies extensions to ~/.pi-extensions/<name> and updates pi's settings.json. Always replaces existing copies.

📝 Auto Documentation

Analyzes source code structure and generates README.md and ARCHITECTURE.md automatically.

🔧 CLI Access

Standalone CLI with --help, --version, --json, and --verbose flags.

🧪 41 Tests

Vitest suite covering validator, installer, and router logic. Pre-commit hook enforces quality.

2. Quick Start

Install the extension-creator itself

git clone <repo-url>
cd pi-extension-creator
npm install
npm run build
npm run bootstrap

The bootstrap command validates the package, copies it to ~/.pi-extensions/pi-extension-creator, and adds it to ~/.pi/agent/settings.json so pi can discover it.

Note: After bootstrapping, restart pi to load the extension.

Verify it's working

# Via CLI
pi-extension-creator validate . --verbose
pi-extension-creator --help

# Via tool (from within pi)
# Call the extension_creator tool with any goal
extension_creator(goal="validate this extension", path=".")

3. Extension Lifecycle

The extension-creator organizes development into seven stages:

💡 idea → 📝 draft → 📁 workspace → ✅ validated → 📦 installed → 🔧 maintenance

ideaJust an idea, no code yetplan
draftStarter scaffold createdscaffold
workspaceExternal repo/folder existsreview
validatedPackage passed validationinstall
installedAlready in pireview or update
maintenanceNeeds updates or removalupdate or remove

The tool automatically infers the stage from the user's goal using keyword matching:

"uninstall"installed
"update", "refresh"installed

none of the aboveidea

4. Using the extension_creator Tool

The extension_creator tool is the primary interface. It accepts the following parameters:

ParameterDescriptionRequired
goalWhat you want to build or doUsually
modeLifecycle action: plan, scaffold, review, validate, install, update, remove, documentAuto-inferred
stageCurrent stage: idea, draft, workspace, validated, installed, maintenanceAuto-inferred
path / sourcePathPath to the extension workspaceRequired for validate/install/update/remove
extensionKindtool, command, prompt, providerAuto-inferred from goal
strictEnforce stricter validation rulesOptional
noteAdditional context or instructionsOptional

How mode is inferred

If you don't specify a mode, the tool infers it in this priority order:

  1. From stage: idea → plan, draft → scaffold, workspace → review, validated → install, installed → review, maintenance → update
  2. From goal keywords: "document" → document, "remove"/"uninstall"/"delete" → remove, "update"/"refresh"/"upgrade" → update, "validate"/"check"/"lint"/"type" → validate
  3. From path: if a path is provided, defaults to review
  4. Fallback: plan

How extension kind is inferred

Goal ContainsInferred Kind
"prompt"prompt/system-prompt extension
"provider"provider extension
"tool"tool extension
"command"command extension
"ui"ui extension
"session" / "state"session/state extension
"resource"resource-discovery extension
"sound" / "notify" / "alert"notification/sound extension

7. How Installation Works

The installer uses deterministic file copy — not symlinks, not npm link.

Source workspace → ~/.pi-extensions/<name> ┌────────────────────┐ ┌──────────────────────┐ │ my-extension/ │ │ my-extension/ │ │ ├── package.json │ ──cpSync──→│ ├── package.json │ │ ├── tsconfig.json │ │ ├── tsconfig.json │ │ └── src/ │ │ └── src/ │ └────────────────────┘ └──────────────────────┘ │ ▼ ~/.pi/agent/settings.json "packages": ["~/.pi-extensions/my-extension"]

Install flow

  1. Validation check: If validation status is fail, install is refused
  2. Name resolution: Uses package.json name (or directory basename as fallback)
  3. Create target: ~/.pi-extensions/ is created if it doesn't exist
  4. Replace existing: If the extension is already installed, it's removed first
  5. Copy files: fs.cpSync(source, installPath, { recursive: true, force: true })
  6. Update settings: Removes old source path from ~/.pi/agent/settings.json, adds install path
Note: The installer logs settings.json errors via console.warn instead of failing. A corrupt settings.json won't block installation.

8. Generating Documentation

The document mode analyzes your extension source code and generates documentation automatically.

What it does

  1. Reads package.json for extension name and description
  2. Resolves the entrypoint source file
  3. Detects the extension kind (tool, command, prompt, provider, complex)
  4. Detects whether the extension has commands, tools, prompts, or skills
  5. Generates README.md with badges, features, install guide, and dev commands
  6. Generates ARCHITECTURE.md (only for complex extensions with many source files)
  7. Writes the files directly to the extension's source directory

Usage

extension_creator(mode="document", path="./my-extension")

9. Workflow Examples

Example 1: Create a new tool extension

User: "Create a tool extension that adds git operations"

Step 1 — Plan
  LLM → extension_creator(goal="git tool extension", mode="plan", stage="idea")
  Tool → Returns architecture plan, files, steps, cautions

Step 2 — Implement
  LLM → Creates workspace directory, package.json, tsconfig.json, entrypoint source

Step 3 — Validate
  LLM → extension_creator(mode="validate", path="./git-tool")
  Tool → Returns pass/warn/fail with diagnostics

Step 4 — Fix (if needed)
  LLM → Addresses errors, re-validates

Step 5 — Install
  LLM → extension_creator(mode="install", path="./git-tool")
  Tool → Copies to ~/.pi-extensions/git-tool, updates settings.json

Step 6 — Document (optional)
  LLM → extension_creator(mode="document", path="./git-tool")
  Tool → Generates README.md + ARCHITECTURE.md

Example 2: Review and update an existing extension

User: "Review my extension in ~/my-extension and update it"

Step 1
  LLM → extension_creator(path="~/my-extension", mode="review")
  Tool → Validates and reports status

Step 2
  LLM → Makes changes based on feedback
  LLM → extension_creator(mode="validate", path="~/my-extension")

Step 3
  LLM → extension_creator(mode="update", path="~/my-extension")
  Tool → Re-validates and updates the installation

Example 3: Using the CLI

# Validate a package
pi-extension-creator validate ./my-extension --verbose

# Get machine-readable results
pi-extension-creator validate ./my-extension --json | jq

# Bootstrap (self-install from current directory)
pi-extension-creator bootstrap

# Get help
pi-extension-creator --help

10. Troubleshooting

Installation fails

SymptomLikely CauseFix
"Validation failed"Package has errorsRun validate mode, check diagnostics, fix errors
"Source path does not exist"Wrong pathUse absolute path or verify the directory exists
Settings.json not updatedPermissions or corrupt JSONCheck ~/.pi/agent/settings.json manually

CLI issues

SymptomFix
"command not found: pi-extension-creator"Run npm run build first, or use npx pi-extension-creator
"Unknown command"Run pi-extension-creator --help for valid commands
TypeScript errors on validationCheck that all imports are correct and tsconfig.json is present

Validation gives unexpected results

  • Entrypoint not found: Ensure pi.extensions in package.json points to the correct path
  • Name not kebab-case: Use lowercase with hyphens: my-extension, not MyExtension or my_extension
  • Warning about index.ts: Prefer named entrypoints like my-tool.ts over index.ts
